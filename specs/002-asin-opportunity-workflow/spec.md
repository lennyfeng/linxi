# Feature Specification: ASIN 新品机会分析工作流

**Feature Branch**: `daiguangsen.windsurf`  
**Spec ID**: `002-asin-opportunity-workflow`  
**Created**: 2026-04-29  
**Status**: Draft  
**Input**: 产品开发设计师在页面填入需要分析的 ASIN 列表，系统通过 sellersprite-mcp 进行产品数据分析，合适的产品标记进入下一轮，下一轮进行产品设计与评估，符合要求后继续进入产品打样等环节。

---

## 1. Product Goal

建立一个面向产品开发设计师和产品经理的 ASIN 新品机会分析工作流，通过批量 ASIN 输入、SellerSprite MCP 数据分析、机会评分、人工复核、设计评估和打样推进，形成从竞品发现到新品立项的标准化漏斗。

该功能的核心价值是：

- 降低人工调研成本
- 统一新品机会判断标准
- 沉淀 ASIN 分析过程和决策依据
- 提升产品开发设计师的选品质量
- 将优秀 ASIN 转化为可设计、可评估、可打样的新产品机会

---

## 2. User Scenarios & Testing *(mandatory)*

### User Story 1 - 批量提交 ASIN 分析任务 (Priority: P1)

作为产品开发设计师，我希望在页面中批量输入 ASIN 列表并提交分析，这样系统可以自动帮我完成竞品数据采集和初步机会判断。

#### Acceptance Scenarios

1. **Given** 设计师进入 ASIN 分析创建页面，**When** 输入多个有效 ASIN 并选择站点后提交，**Then** 系统创建一个分析批次并展示分析进度。
2. **Given** 输入内容中包含重复 ASIN，**When** 提交，**Then** 系统自动去重并提示重复数量。
3. **Given** 输入内容中包含 Amazon 商品链接，**When** 提交，**Then** 系统自动提取链接中的 ASIN。
4. **Given** 输入内容中包含非法 ASIN，**When** 提交，**Then** 系统阻止提交或标记非法项，并显示具体错误。

---

### User Story 2 - 查看 SellerSprite MCP 分析结果 (Priority: P1)

作为产品开发设计师，我希望查看每个 ASIN 的商品数据、市场数据、竞争数据、流量数据和系统推荐结论，这样我可以判断是否进入下一轮设计评估。

#### Acceptance Scenarios

1. **Given** ASIN 分析任务已完成，**When** 用户进入分析结果页，**Then** 系统展示每个 ASIN 的图片、标题、价格、评分、Review 数量、销量、销售额、BSR、机会分和推荐结论。
2. **Given** 某个 ASIN 分析失败，**When** 用户查看结果，**Then** 系统展示失败原因并提供重新分析操作。
3. **Given** 用户点击某个 ASIN 的详情，**When** 打开分析报告，**Then** 系统展示商品基础信息、销量趋势、关键词流量、竞品结构、评价痛点、风险和机会分析。
4. **Given** MCP 返回原始数据，**When** 分析完成，**Then** 系统保存标准化字段和原始数据快照。

---

### User Story 3 - 标记进入下一轮或淘汰 (Priority: P1)

作为产品经理或设计师，我希望基于系统评分和人工判断，将合适的 ASIN 标记进入下一轮，将不合适的 ASIN 淘汰，并记录原因。

#### Acceptance Scenarios

1. **Given** ASIN 分析完成且推荐进入下一轮，**When** 用户点击“进入下一轮”，**Then** 状态变为“待设计评估”。
2. **Given** ASIN 不符合要求，**When** 用户点击“淘汰”，**Then** 系统要求填写淘汰原因并记录审核历史。
3. **Given** 系统推荐淘汰但用户人工判断仍有机会，**When** 用户强制进入下一轮，**Then** 系统要求填写 override 原因。
4. **Given** 已淘汰 ASIN，**When** 普通设计师查看，**Then** 只能查看历史记录，不能继续推进。

---

### User Story 4 - 产品设计与差异化评估 (Priority: P2)

作为产品开发设计师，我希望对进入下一轮的 ASIN 填写设计评估，包括目标用户、使用场景、痛点、改良方向、材料、结构、包装和卖点，这样产品经理可以审核是否具备开发价值。

#### Acceptance Scenarios

1. **Given** ASIN 状态为“待设计评估”，**When** 设计师填写设计评估并提交，**Then** 状态变为“设计方案待审核”。
2. **Given** 设计评估需要附件，**When** 设计师上传参考图、草图或设计说明，**Then** 系统保存附件并在详情页展示。
3. **Given** 产品经理审核设计方案，**When** 点击通过，**Then** 状态进入“待成本与打样评估”。
4. **Given** 产品经理认为方案不完整，**When** 点击驳回，**Then** 系统要求填写驳回原因，状态回到“待设计评估”。

---

### User Story 5 - 成本与打样评估 (Priority: P3)

作为采购或产品经理，我希望对设计通过的 ASIN 进行成本、供应链和打样可行性评估，这样可以判断是否进入打样阶段。

#### Acceptance Scenarios

1. **Given** 设计方案已通过，**When** 采购填写目标售价、预估成本、MOQ、供应商、打样费用和周期，**Then** 系统保存成本评估。
2. **Given** 成本评估通过，**When** 产品经理点击“允许打样”，**Then** 状态变为“打样中”。
3. **Given** 成本过高或供应链不可行，**When** 产品经理点击“终止”，**Then** 系统记录终止原因并结束流程。

---

## 3. Edge Cases

- ASIN 属于不存在或已下架商品
- SellerSprite MCP 调用超时、限流或返回空数据
- 同一个 ASIN 被多个批次重复提交
- 不同 marketplace 下相同 ASIN 数据不同
- 商品存在变体，当前 ASIN 与父 ASIN 数据不一致
- 商品销量高但评价极差
- 商品评分高但 Review 门槛过高
- 市场需求高但品牌/卖家高度垄断
- 用户人工结论与系统推荐冲突
- 设计评估提交后需要补充资料
- 打样通过后需要转入正式新品开发项目

## 4. Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须支持创建 ASIN 分析批次。
- **FR-002**: 系统必须支持批量输入 ASIN，每行、逗号和空格分隔均可识别。
- **FR-003**: 系统必须支持从 Amazon 商品链接中自动提取 ASIN。
- **FR-004**: 系统必须校验 ASIN 格式、去重，并标记非法 ASIN。
- **FR-005**: 系统必须支持选择 marketplace，默认 US。
- **FR-006**: 系统必须为每个 ASIN 创建独立分析项。
- **FR-007**: 系统必须通过 sellersprite-mcp 获取 ASIN 商品基础信息。
- **FR-008**: 系统必须通过 sellersprite-mcp 获取销量、销售额、BSR 和价格趋势。
- **FR-009**: 系统必须通过 sellersprite-mcp 获取关键词和流量结构数据。
- **FR-010**: 系统必须通过 sellersprite-mcp 获取竞品和关联商品数据。
- **FR-011**: 系统必须通过 sellersprite-mcp 获取用户评论数据，用于痛点分析。
- **FR-012**: 系统必须保存 MCP 原始返回数据快照。
- **FR-013**: 系统必须将 MCP 数据标准化为可筛选字段。
- **FR-014**: 系统必须为每个 ASIN 生成市场分、竞争分、设计机会分、风险分和综合机会分。
- **FR-015**: 系统必须根据评分生成推荐结论：推荐进入下一轮、需人工复核、建议淘汰。
- **FR-016**: 系统必须支持分析失败重试。
- **FR-017**: 系统必须展示批次维度的分析进度和统计。
- **FR-018**: 系统必须支持 ASIN 结果列表的搜索、筛选、排序和分页。
- **FR-019**: 系统必须支持查看单个 ASIN 的完整分析报告。
- **FR-020**: 系统必须支持用户将 ASIN 标记为进入下一轮。
- **FR-021**: 系统必须支持用户淘汰 ASIN，并强制填写淘汰原因。
- **FR-022**: 系统必须支持人工 override 系统推荐，并记录原因。
- **FR-023**: 系统必须记录所有状态流转和人工决策历史。
- **FR-024**: 系统必须支持填写产品设计评估。
- **FR-025**: 系统必须支持上传设计附件。
- **FR-026**: 系统必须支持设计方案审核通过、驳回和补充资料。
- **FR-027**: 系统必须支持成本与打样评估。
- **FR-028**: 系统必须支持将样品通过的机会转入正式新品开发项目。
- **FR-029**: 系统必须按角色控制操作权限。
- **FR-030**: 系统必须支持导出 ASIN 分析结果。

### Non-Functional Requirements

- **NFR-001**: 单个 ASIN 分析任务失败不得影响同批次其他 ASIN。
- **NFR-002**: MCP 调用必须异步执行，页面不得长时间阻塞。
- **NFR-003**: 列表页 1000 条以内数据分页查询响应时间应小于 500ms。
- **NFR-004**: 所有关键操作必须记录操作者、时间和原因。
- **NFR-005**: 系统必须可追溯历史分析数据，不因后续 MCP 数据变化覆盖历史决策依据。

## 6. Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 产品开发设计师可以在 3 分钟内完成 20 个 ASIN 的批量录入与提交。
- **SC-002**: 系统可以展示每个 ASIN 的分析状态，用户无需刷新页面即可知道任务是否完成。
- **SC-003**: 至少 90% 的有效 ASIN 能生成基础分析报告。
- **SC-004**: 用户可以基于系统推荐在 30 秒内完成单个 ASIN 的进入下一轮或淘汰决策。
- **SC-005**: 每个进入设计评估的 ASIN 都能追溯到原始分析数据和人工决策原因。
- **SC-006**: 管理者可以看到批次级别的推荐率、淘汰率和进入设计评估数量。
- **SC-007**: 系统支持从 ASIN 录入到设计评估提交的完整 MVP 流程。

---

## 7. Assumptions

- 当前系统已具备登录、权限、用户、菜单和数据库基础能力。
- SellerSprite MCP 可由后端服务调用或由 AI Agent 辅助调用并写入分析结果。
- MVP 阶段优先实现 ASIN 分析到设计评估，不强制完整实现供应商报价和样品质检闭环。
- marketplace 初期以 US 为主，后续扩展 UK、DE、JP 等站点。
- 产品设计附件可先使用现有 MinIO 配置或以 URL/JSON 方式保存。
- 系统推荐结论只作为辅助决策，最终决策以人工操作为准。
