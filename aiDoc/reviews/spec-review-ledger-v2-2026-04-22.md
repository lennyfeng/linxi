## 二、宪法合规性检查

### 🔴 原则 5「分阶段交付闭环」— 范围膨胀风险

**问题**: 两份 Spec 合计包含 **9 个 FR + 4 个页面重构**，涉及：
- 2 张新建表 (ledger_projects, ledger_counterparties)
- 1 个新基础设施对接 (MinIO 上传)
- 1 个新前端依赖 (@dnd-kit)
- 4 个页面完整重构
- 多维度汇总后端 API（可能需要 10+ 个新 SQL 查询）

这不是一个"最小闭环"，而是一次**全面翻新**。宪法要求"如果需求显著增加复杂度，默认推迟到下一阶段"。

**建议**: 必须拆分为 2-3 个交付批次：
- **Batch 1（核心体验修复, ~3天）**: FR-2 布局合并、FR-3 中文、FR-4 搜索修复、FR-6 币种隐藏、FR-7 分类层级、操作按钮（流水行编辑/复制/删除）
- **Batch 2（管理页 CRUD, ~5天）**: 项目管理 CRUD + 商家管理 CRUD + 账户管理补完
- **Batch 3（高级功能, ~3天）**: 多维度汇总切换、拖拽排序、图片上传

### 🟡 原则 2「单一数据源」— 新表与文本字段的冲突

**问题**: 当前 `transactions.project_name` 和 `transactions.counterparty_name` 是纯文本字段。Spec 提出新建 `ledger_projects` 和 `ledger_counterparties` 管理表，但关联方式仍然是 **写 name 到文本字段**。

这意味着：
- 项目改名后，历史流水的 project_name 不会同步更新 → **数据不一致**
- 同名项目/商家无法区分
- 删除项目后，历史流水的 project_name 变成孤立数据

Spec 虽然提到"保持向后兼容"，但没有说明：
1. 改名时是否要联动更新历史流水？
2. 是否要加 project_id / counterparty_id 外键？
3. 孤立数据如何处理？

**建议**: 明确以下规则并写入 Spec：
- **方案 A（推荐）**: 新增 `transactions.project_id` 和 `counterparty_id` 外键字段，保留 name 字段作为冗余快照。查询用 id 关联，显示用 name。改名时更新 name 快照（可异步）。
- **方案 B（简单）**: 继续用文本字段，项目/商家改名时批量 UPDATE 历史流水。删除项目时不删历史流水的 name。明确说明这是过渡方案。
- 必须选定一个并在 Spec 中标注。

### 🟢 原则 7「复用现有模式」— 新依赖 @dnd-kit 的必要性

**问题**: Spec 提议引入 `@dnd-kit` (3个包) 做拖拽排序。目前项目已有 Ant Design，其 Table 可通过 `react-dnd` 或内置的拖拽行实现基本排序。

**建议**: 先确认 Ant Design Table 自带的拖拽行（`components` 自定义 + `react-dnd`，已有很多成熟示例）是否满足需求，避免引入全新依赖栈。如果 @dnd-kit 确实更好，需要在 Spec 中说明理由。

## 三、FR 逐项评审

### FR-1: 左侧面板 + 多维度切换
🔴 **范围过大，缺乏优先级排序**

- "180px 或可拖拽调整" — 拖拽调整面板宽度是一个独立功能，不应夹在 FR-1 里
- "支持切换维度（时间>年/季/月/周/天、分类>一级/二级、账户、项目、其它>成员/商家/记账人）" — 这需要 **至少 12 个新后端 API**。FR-1 说"涉及文件"只列了前端文件，但后端的工作量远大于前端
- "中部可选择一个类目进行筛选" — 这个交互描述模糊。是在表头加下拉？还是在筛选栏新增一个选择器？

**建议**:
- Phase 1 只做 **缩窄 + 中文**，维度切换延后
- 维度切换需要独立评估后端 API 工作量（每个 groupBy 维度可能需要不同的 SQL）
- "中部类目筛选"需要补充交互原型或更清晰的文字描述

### FR-2: 汇总 + 按钮合并
🟢 **合理，细节待完善**

- 截图中按钮包括"批量导入"、"筛选"、"批量编辑" — 但 Spec 列的是"记一笔、搜索框、筛选、刷新"。按钮列表不一致。
- 需要明确：哪些按钮是常驻的，哪些放到"更多"下拉里？

### FR-3: 中文月份
🟢 **简单，无风险**

- 建议方式 A（dayjs locale），因为全局生效，一劳永逸

### FR-4: 搜索修复
🟡 **缺失性能评估和边界规则**

- "金额匹配：keyword 是数字时额外匹配 amount" — 如果用户输入 `100`，会匹配所有金额 = 100 的流水 + 所有包含 "100" 的摘要/备注。结果可能非常多。
- LIKE 搜索 JOIN accounts + categories 在流水量大时可能有性能问题
- 需要补充：搜索结果是否跨月（绕过左侧月份筛选）？还是仅在当前选中月份范围内搜索？

**建议**:
- 明确搜索范围（当前月份 vs 全部）
- 金额匹配考虑精确匹配 + 前缀匹配（如 `100` 匹配 100.00，而非 1001.00）
- 添加索引评估（counterparty_name, project_name 是否需要加索引）

### FR-5: 管理页统一改造（指向独立 Spec）
见下方"管理页 Spec 评审"。

### FR-6: 记一笔 — 隐藏币种/汇率
🟡 **汇率处理逻辑需要补充边界**

- "后端自动查询当日汇率" — exchange_rates 表的数据来源是什么？手工录入还是自动拉取？如果某天没有汇率怎么办？
- "使用最近一次汇率并标记" — 标记在哪里？流水列表上是否显示"汇率已过期"？
- 转账场景（两个不同币种账户之间）：两端汇率如何处理？

**建议**:
- 补充 exchange_rates 的数据来源和更新机制说明
- 汇率过期（>N天）时前端应提示用户确认
- 转账场景需要单独描述汇率处理流程

### FR-7: 分类层级选择
🟢 **合理**

- 建议使用 TreeSelect，因为与项目选择器的交互模式一致（都是树形选择）
- "如果分类没有子分类，一级分类本身可选" — 这个规则很重要，需要在实现时确保

### FR-8: 项目下拉
🟡 **"支持用户输入新项目名"有隐患**

- "如果列表没有匹配的" 允许直接输入 → 这意味着可以绕过 ledger_projects 管理表，直接创建"野生"项目名
- 这与 FR-5 的项目管理逻辑矛盾：花精力建管理表和三级树形，但记账时可以随意输入未管理的项目名

**建议**:
- **要么严格**: 只允许从 ledger_projects 下拉选择，不允许自由输入。新项目必须先在项目管理页创建。
- **要么宽松**: 允许自由输入，但输入新项目名时弹出提示"项目 XXX 不在管理列表中，是否同时创建？"
- 必须选定一个策略，不能两头放任。

### FR-9: 图片上传
🟡 **MinIO 集成细节不足**

- MinIO bucket 名称、权限策略、文件命名规则未说明
- 缩略图生成策略：上传时生成？还是前端用 CSS 缩放原图？
- 大文件限制：最大文件大小？支持的格式？
- 删除附件时是否同步删除 MinIO 上的文件？

**建议**:
- 补充 MinIO bucket 配置（名称、ACL、文件路径规则）
- 补充文件大小限制（建议 5MB/张）和格式限制（jpg/png/pdf）
- 删除策略：先 DB 软删除，定期清理 MinIO（或立即删除）

## 四、管理页 Spec (ledger-management-pages-v2.md) 评审

### 🔴 账户管理「分组方式」设计有误

**问题**: Spec 说"按 type 分组作为一级（现金账户、储蓄账户、虚拟账户等），子账户为二级"。但截图中的分组是：
- 现金账户
- **储蓄账户**（截图里有，但 DB 的 type ENUM 没有 `savings`）
- **虚拟账户**（截图里有，但 DB 的 type ENUM 没有 `virtual`）

当前 DB 的 type 枚举是 `bank, cash, alipay, wechat, credit_card, other`。截图中的分组名（现金账户/储蓄账户/虚拟账户）与 DB 枚举不匹配。

**建议**:
- 需要确认：分组到底是按 DB 的 type 字段，还是新增一个 `account_group` 字段？
- 如果用现有 type 字段，需要更新 ENUM 值为更合理的分组（如 `cash`, `savings`, `virtual`, `credit`）
- 或者新增 `account_group` VARCHAR 字段，与 type 解耦

### 🟡 商家管理 — 树形结构的必要性存疑

**问题**: 商家管理引入 parent_id 做树形分组。但商家是否真的需要层级？通常商家是扁平的。

**建议**:
- 先确认实际业务场景：商家是否有"母公司 → 子公司"这种关系？
- 如果只是为了分组展示（如"平台服务商"分组），可以用 `group_name` 字段替代树形，实现更简单
- 不要为了"统一"而给不需要树形的实体强加树形结构

### 🟡 分类管理 — Tree 改 Table 的必要性

**问题**: 当前分类管理用 `Tree` 组件，展示效果其实挺好（树形结构天然适合 Tree 组件）。改为 Table + expandable 主要是为了"与其他页面统一"，但分类的交互场景（层级浏览 + 快速增删）与账户/项目不同。

**建议**:
- 保留 Tree 组件未必不行，关键是补齐缺失功能（删除、拖拽排序）
- Tree 组件也支持拖拽（Ant Design `Tree` 有 `draggable` prop）
- 如果为了统一而改 Table，需要确保 Table 的交互不比 Tree 差

### 🟢 排序 API — 设计合理

统一的 `PUT /ledger/{entity}/sort` 接口设计简洁一致。校验 parent_id 一致性是必要的安全措施。

## 五、缺失项（两份 Spec 都遗漏的）

### 🔴 1. 删除流水的影响范围未评估

FR-5（管理页 Spec）写了管理实体的删除，但**流水行的删除**（FR-2 提到操作按钮有删除）没有说明：
- 已提交（submitted）的流水能否删除？
- 删除后是否影响账户余额（current_balance 需要回退）？
- 删除是软删除还是硬删除？
- 是否需要记录操作日志（constitution 原则 6: 可追溯性）？

**这是业务关键问题，必须在 Spec 中明确。**

### 🟡 2. 缺少错误状态和边界处理描述

- 汇率缺失时的 UI 表现
- 搜索无结果时的 Empty State
- 拖拽排序失败时的回滚策略
- MinIO 不可用时的降级方案（是否阻塞记账？）
- 网络中断时的保存策略（draft 自动保存？）

### 🟡 3. 缺少数据迁移方案

新建 ledger_projects 和 ledger_counterparties 后：
- 现有 transactions 里的 project_name 和 counterparty_name 是否需要迁移到新表？
- 如果不迁移，管理页上看不到历史数据的项目/商家，用户会困惑
- 建议增加一次性迁移脚本：从 transactions 中 extract distinct project_name/counterparty_name → 批量插入新表

### 🟢 4. 权限控制未提及

虽然 Spec 说"无额外权限变更"，但删除操作（流水删除、账户删除、项目删除）是否需要限制角色？至少应该说明"当前不做权限限制，后续再加"。
