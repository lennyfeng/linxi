> Reviewing `speckit-specify.md` and `development-plan.md` from 4 roles.
> Date: 2026-04-21

## 1. CEO / 老板视角 — 跨境电商公司经营者

### Core concerns
商业价值、投入产出比、竞争力、决策支持、风险控制

### 🔴 严重遗漏

#### 1.1 缺少经营决策核心数据
现在的 Dashboard 只是"我的待办 + 财务快照"，对老板基本没用。跨境电商老板每天要看的是：
- **多平台销售汇总**：Amazon (按站点)、TikTok Shop、独立站、Shopee 等各渠道销售额/订单量/客单价
- **毛利率 & ROAS**：按 SKU / 品类 / 店铺 / 时间维度的盈利能力
- **广告花费 & ACOS**：广告投产比，占销售额的百分比
- **FBA 库存健康度**：库存周转天数、滞销 SKU、断货预警、补货建议
- **资金周转**：账期、在途资金、可用资金、未来 30/60/90 天现金流预测
- **Listing 健康度**：差评率、退货率、Buy Box 占有率、链接抑制情况

当前规格完全没有这些，**Dashboard 需要重新设计**。

#### 1.2 缺少店铺/平台维度
整个系统默认单店铺单平台，但跨境电商典型结构是：
- 多国站点（US / UK / DE / JP / CA ...）
- 多平台（Amazon / TikTok Shop / 独立站 / Shopee / Temu）
- 多主体（中国公司 + 香港公司 + 美国公司）

**规格中没有店铺/平台/主体三级维度**，所有数据都是"单一视角"，实际业务无法运转。

#### 1.3 Lingxing 单点依赖风险
- 所有业务数据依赖 Lingxing，Lingxing 换 API 或涨价我们就瘫痪
- 建议：Lingxing 数据要本地存档（已做），还需要**双向写回能力**做备份
- 关键数据（订单、库存）建议直接对接 Amazon SP-API，减少对领星的依赖

#### 1.4 广告投放数据缺失
跨境电商成本占比：产品成本 40% + 物流 15% + 平台佣金 15% + **广告 15-25%** + 其他。
广告是仅次于产品成本的第二大开支，但规格里**完全没有广告数据**。
至少要接入：Amazon Ads API（获取广告花费、ACOS、转化、关键词）

#### 1.5 FBA/退货管理缺失
跨境电商的独特痛点，现有规格**零覆盖**：
- FBA 入库/在途/可售/滞销/不可售库存分层
- 退货率监控（Amazon 退货 > 10% 链接会被抑制）
- 客户返修/售后流程

### 🟡 改进建议

- 增加 **"经营分析"模块**（Module 6）：KPI 大屏、平台/店铺/SKU 多维穿透
- 增加 **"店铺管理"基础数据**：店铺、平台、主体、市场、币种、账期规则
- Dashboard 分角色：老板看经营大屏，财务看财务快照，产品看项目进度

### ROI 评估
14 周开发 + 维护成本，**内部工具回本关键在能否替代现有 SaaS 订阅**。
- 如果当前在用 SaaS 预算：需量化（随手记会员费、其他工具费）
- 如果此系统能整合 SaaS 分散数据，节省决策时间：价值显著
- **建议加上成功指标**：比如"老板每天看报表从多个系统减少到 1 个，耗时从 2h → 15min"

## 2. Product Director / 产品总监视角 — 新品开发 & 管理

### Core concerns
新品成功率、开发速度、选品数据、竞品跟踪、Listing 质量

### 🔴 严重遗漏

#### 2.1 选品阶段数据支撑为零
"Research 研究"阶段只是一个存图片/链接的空壳，缺：
- **关键词挖掘**：Helium10 / Jungle Scout / 卖家精灵数据（搜索量、竞争度、相关词）
- **市场容量估算**：类目销售额、头部产品销量
- **竞品跟踪**：指定 ASIN → 自动拉取 BSR、Review 数、价格、上架时间变化曲线
- **合规预检**：关键词、图片版权、专利、商标、CPSC/CE/FCC 认证要求

没有这些数据，"选品"依赖拍脑袋，新品成功率不会提升。

#### 2.2 多渠道利润试算被明确放到 Phase 2
规格里"Non-Amazon profit calculation"列为 NOT do，但：
- TikTok Shop 的佣金结构（5% 平台费 + 1.5% 服务费 + 物流补贴 + 达人分佣）完全不同
- Temu / SHEIN 的全托管/半托管模式成本结构又不同
- 独立站要算 Shopify 费 + 信用卡费 + 广告费 + 物流

**实际选品时就需要多渠道对比**，Phase 1 只有 Amazon 会导致选出来的品不能投 TikTok。建议 Phase 1 至少加 TikTok Shop 公式。

#### 2.3 Listing 管理缺失
新品同步到 Lingxing 后就"完结"了，但真正的工作才开始：
- Listing 优化（标题 / 五点 / 描述 / A+ / 后台搜索词）
- Listing 版本管理与审批
- 多语言翻译工作流
- 上架后 30 天 Review 数量跟踪

#### 2.4 新品生命周期只跟到"首单"
规格里流程终点是 First Order Placed，但产品总监关心的是：
- **上架后 30/60/90 天表现**：Review、BSR、转化率
- **产品生命周期阶段**：新品期 → 稳定期 → 衰退期 → 清仓/下架
- **首单后补货决策**：基于销售速度和 FBA 库存的补货建议

建议生命周期延长：首单 → 上架 → 跟踪期 → 成熟期 → 衰退期

#### 2.5 成本管理过于简化
- 没有 **BOM / 配件管理**（规格里 NOT in Phase 1）— 但跨境产品常有配件/包材/说明书多个 SKU 组成
- 没有 **认证/合规费用** 分摊
- 没有 **开模/打样/测试费用** 跟踪

### 🟡 改进建议

- 利润试算 Phase 1 必须支持 **至少 Amazon + TikTok Shop 两个渠道**
- 增加 **What-if 分析**：价格敏感度、成本敏感度、FBA 费敏感度滑块
- 增加 **竞品跟踪模块**：定时抓取 Amazon 产品页数据（可通过 Keepa API）
- 审批流程太重，建议 **按金额分级**：<1 万元快速审批（只需产品负责人），>5 万元完整 4 节点

### 协作功能缺失
规格里 "comments / @mentions (future)" 被推迟，但跨团队协作严重依赖讨论。建议 Phase 1 就做 **基础评论 + @ 通知**，成本不高但价值大。

## 3. Finance Director / 财务总监视角 — 财务合规 & 报表

### Core concerns
准确性、合规性、审计、现金管理、税务

### 🔴 严重遗漏

#### 3.1 单式记账的局限性必须明确
规格写了"single-ledger"，这意味着本系统只是**管理报表系统**，不是**法定会计系统**。
- ⚠️ 合规账由柠檬云处理，需明确两套账如何对接
- 需要标注：本系统数据**不能作为审计依据**，只能作为经营参考
- 如果未来审计需要，需预留导出到会计系统的接口

#### 3.2 外币重估 & 汇兑损益缺失
规格里"exchange gain/loss auto-calculated"但**公式未定义**：
- 跨币种转账时点汇率差
- **期末外币账户重估**：账户余额按期末汇率 vs 账面汇率的差
- 外币应收应付的重估

这是跨境电商财务的核心业务，**必须明确算法**。

#### 3.3 应收应付管理缺失
- 账期管理：Amazon 14 天回款、TikTok 月结、客户月结
- 应收账龄分析
- 应付账龄分析
- 供应商账期 & 到期预警

#### 3.4 现金流预测缺失（跨境电商命根子）
跨境电商**现金转换周期**长：采购出钱 → 发货 → 上架 → 销售 → 回款，平均 60-90 天。
财务总监每天最关心的是：
- 未来 30 / 60 / 90 天**预计流入**（Amazon 应收、TikTok 应收、客户回款）
- 未来 30 / 60 / 90 天**预计流出**（采购请款、工资、广告预存、物流预付）
- **净现金流缺口预警**：提前知道哪天会缺钱

现有规格完全没有预测能力，这是**最大的财务价值缺失**。

#### 3.5 税务处理空白
- 增值税进项/销项分账
- 出口退税
- 跨境电商综合税（针对独立站和 TikTok）
- **9610 / 9710 / 9810 报关单据**对接

跨境电商的税务规则比国内企业复杂得多，规格里一个字没有。

#### 3.6 关键报表缺失
现在只有"income/expense summary + account balances"，远远不够。至少需要：
- **利润表** (P&L)：按月/季/年，按店铺/平台/主体
- **SKU 盈亏分析**：每个 SKU 真实毛利（含分摊广告、分摊物流、分摊退货）
- **店铺盈亏分析**：Amazon US vs UK vs DE 各自的真实利润
- **广告 ROI 报表**

### 🟡 期初余额问题

规格说 "all accounts start at 0"。
- 如果公司已运营多年，当前所有账户有真实余额
- 以 0 开始 = 账面资产为 0，所有历史导入进去会 **扭曲财务状态**
- 建议两选一：
  - 方案 A：以某个日期切换，之前的用"期初调整"一次性录入实际余额
  - 方案 B：把随手记**全部历史**都导入（数据清洗脚本已规划），账户余额自然累计出来

规格选了方案 B（"full history imported"），但随手记最早能追溯到什么时候？如果公司比随手记账本还早，方案 B 也有漏洞。**需要确认公司成立至今所有资金流是否都在随手记**。

#### 3.7 Airwallex 对账 ≠ Airwallex 导入
现在只做了导入，但对账意味着：
- Airwallex 每笔收款对应到销售订单
- 手续费对应到费用科目
- 真正的**银行对账表**能导出给会计

### 🟡 改进建议
- 明确本系统与柠檬云的**分工边界**文档
- Phase 2 加入基础 **现金流预测**（即使先简化：按历史同期估算）
- Phase 2 加入 **SKU 盈亏分析**
- 规格中加入 **报表清单附录**

## 4. Dev Manager / 开发经理视角 — 技术可行性 & 质量

### Core concerns
工期可行性、技术债务、可维护性、团队能力、安全

### 🔴 严重问题

#### 4.1 工期严重低估
14 周单人完成这个规模，**不现实**。
对比估算：
- 基础设施 + 用户权限：3 周（保守）
- 记账 + 随手记迁移：4 周（含 PDF 解析 + 多币种 + 报表 + Python 迁移脚本）
- 勾稽：4 周（4 类数据同步 + 多对多关系 + 推荐 + 报表）
- 新品开发：4 周（看板 + 多步表单 + 利润试算 + 审批 + Lingxing 同步）
- 横切 + 部署 + 测试：3 周

**合理估算：18-22 周单人**，如果要加上前面 3 个视角的缺失项 → **30+ 周**。

**建议：**
- 立即澄清是单人还是团队
- 按模块分包并行（前后端各 1 人、全栈 2 人 → 12-14 周可达成）
- 或者 **明确 Phase 1 MVP 范围**，先上线核心功能

#### 4.2 技术栈老化/裸奔问题

| 现状 | 问题 | 建议 |
|------|------|------|
| 裸 Node HTTP server | 路由/中间件/错误处理全手写，易出 bug | 迁移到 **Fastify**（性能好，TS 友好）或保留现状但完善 |
| 无 ORM（raw mysql2） | SQL 注入风险、类型不安全、重构困难 | 引入 **Drizzle ORM**（轻量，TS 原生）或 Prisma |
| 无测试框架 | 只有 regression .mjs 脚本 | 加 **Vitest** 单元测试 + **Playwright** E2E |
| 无日志系统 | console.log 生产环境不可用 | 加 **Pino** 结构化日志 |
| 无 API 文档 | 前后端沟通靠口传 | 生成 **OpenAPI** spec + Swagger UI |
| 无数据库迁移工具 | schema 变更无版本控制 | 引入 **node-pg-migrate** 或 **Drizzle migrations** |
| 无错误监控 | 线上报错发现不及时 | 可选：**Sentry** 开源版自建 |

#### 4.3 安全漏洞

| 问题 | 风险 | 方案 |
|------|------|------|
| 领星 AppSecret 存 DB 是否加密？ | 数据库泄露 = 全部凭证泄露 | 用 AES-256 加密 + 密钥环境变量 |
| 登录无限制 | 暴力破解 | 加 **登录限流**（5 次失败锁定 15 分钟） |
| 无 CSRF 保护 | 跨站请求伪造 | API 要求 `SameSite=Strict` Cookie 或自定义 Header |
| 无 CSP | XSS 注入风险 | Nginx 配 Content-Security-Policy |
| 文件上传无扫描 | 恶意文件上传 | 用 **ClamAV** 做扫描 + MIME/扩展名双重校验 |
| JWT 存 localStorage | XSS 可偷 token | 改用 **httpOnly Cookie** + CSRF Token |

#### 4.4 pg-boss / MySQL 冲突暴露架构决策仓促
规格中确认了 pg-boss，但实际数据库是 MySQL。这不是小问题——说明**架构决策没有同步验证**。
建议在 P0 阶段 **强制进行架构评审**，所有技术选型要在动代码前锁定。

#### 4.5 部署 & 运维空白
- 没有 **CI/CD 管道**（GitHub Actions / 自建 Jenkins？）
- 没有 **Staging 环境**（测试服 192.168.1.251 是开发 or 测试？生产是阿里云？）
- 没有 **备份策略**（MySQL dump? MinIO 备份? 频率？保留多久？）
- 没有 **监控告警**（API 宕机、DB 宕机、MinIO 宕机如何第一时间知道？）
- 没有 **灾难恢复**计划

#### 4.6 数据规模估算缺失
- 领星采购单每年可能 1-5 万单 → 3 年 = 15 万条，关联的 SKU 行项目 × 5-20 倍 = 300 万行
- 交易流水每年可能 5-10 万条
- 产品 SKU 数千个

这些规模下，**需要**：
- 表分区 / 索引策略
- 历史数据归档策略
- 报表查询走独立只读库或宽表

规格里**没有任何性能设计**。

### 🟡 流程建议
- **先写 API 合约再写代码**：用 OpenAPI 定义前后端先达成一致
- **前后端分工**：如果只有 1 个全栈 → 按模块垂直切分；如果 2+ 人 → 前后端分工，共享 `packages/shared` 类型
- **代码评审**：每个 Phase 结束前必须 code review + demo
- **渐进式发布**：P1 完成就先上线给用户用（用户权限可独立运行），P2-P4 逐步加功能

## 综合结论 & 推荐动作

### 🔴 需求层面重大缺口（优先补充）
| # | 缺口 | 来源 | 建议 |
|---|------|------|------|
| 1 | 店铺/平台/主体多维度基础数据 | CEO | 必须加入 P0 Schema |
| 2 | Amazon Ads / 广告数据 | CEO/Finance | Phase 2 或独立 Phase |
| 3 | FBA 库存 & 退货管理 | CEO/Product | 独立 Phase 6 |
| 4 | 多渠道利润试算（至少 +TikTok） | Product | Phase 1 扩容 |
| 5 | 现金流预测 | Finance | Phase 2 加入基础版 |
| 6 | SKU 盈亏分析 | Finance | Phase 5 报表模块扩容 |
| 7 | 外币重估公式 & 汇兑损益算法 | Finance | Phase 2 明确 |
| 8 | 竞品跟踪（基础版） | Product | Phase 4 考虑 |
| 9 | 业务分工边界（本系统 vs 柠檬云） | Finance | 文档明确 |
| 10 | 应收应付账期管理 | Finance | Phase 3 勾稽模块扩容 |

### 🔴 计划层面重大调整
| # | 问题 | 调整 |
|---|------|------|
| 1 | 14 周不现实 | 明确人力，合理的话 18-24 周 或 1+ 人团队 |
| 2 | 技术栈裸奔 | P0 补充 ORM / 测试 / 日志 / 迁移工具 |
| 3 | 安全基线缺失 | P0 加入安全清单（加密、限流、CSP、文件扫描） |
| 4 | 运维缺失 | P0 规划 CI/CD、备份、监控 |
| 5 | 数据规模未估算 | P0 加入性能与存储设计 |
| 6 | 分阶段发布策略 | 考虑 MVP 先行，P1 完成即可上线使用 |

### 推荐下一步
在开始 P0 前，**建议先做**：
1. **确认人力**：几个开发？全栈 or 分工？
2. **范围收敛**：上面 10 个重大缺口哪些必须 Phase 1，哪些可以 Phase 2+？
3. **MVP 定义**：最小可上线范围是什么？（我的推荐：用户+记账 2 个模块先上）
4. **架构评审**：ORM 选型、测试框架、日志、部署方案锁定
5. **安全基线**：列清单，P0 内完成

**不建议直接按当前计划开工**，缺口补上后再动。
